#+TITLE: RAG Evaluation

* Overview
Conducted a RAG system evaluation with these parameters changed:
- document splitters; size-based, context-based
- embedding models; all-minilm, nomic, qwen3 0.6b
- LLM models; llama3.1:8b, gemini-2.5-flash-lite

Key takeaways include:
- As a splitter, context-based with sub-splitting large sections was optimal
- Embedding model matters.  qwen3 beat all-minilm and nomic as expected
- Question transformation didn't improve the numbers

* Tests
** Parameters
- Chunk splitters
  - Size-based
    - Size: specify chunk size in number in characters
    - Overlap: how many characters are overlapped between 2 continuous chunks
  - Context-based: split by .org headers (ie, startswith('*'))
  - Context-based + sub-split large sections
- top_k: how many chunks similar to a query are retrieved
- embedding model: all-minilm, nomic-embed-text or qwen3-embedding:0.6b
- answer model:  gemini-2.5-flash-lite or lama3.1:8b
- eval model: gemini-2.5-flash-lite or lama3.1:8b
- question transformation: ask LLM to transform the question to be more vector DB friendly (ie, remove noise and add keywords that LLM knows)

** Questions
- Asked Claude to generate questions.  Prompt used:

Generate your answer based only on the document below, not from your general knowledge. The doc:
#+end_src
- Checked and updated manually
  - "expected_answer" -> "reference_answer"
  - changed some keywords (eg, not appear in data files)
  - asked Claude to re-generate 3 incorrect questions

** Scorers
I used four scorers, two to see retrieval quality and two for end-to-end quality.
*** Retrieval
- Recall@K - percentage of keywords appear in top_k chunks
  - Used approximation with keywords
- MRR (Mean Reciprocal Rank) - how early in (top_k) chunks any keyword appear

*** End-to-end
- Faithfulness - ask LLM to rate if answer is generated solely from top_k chunks
- Answer correctness - ask LLM if the answer answers question

* Takeaways (to be updated)

** Embedding
- size=600 & overlap=60 is the sweet spot
- Context-based splitter with sub-splitting large sections is better than context-based only or size/overlap only
- Embedding model matters.  qwen3-embedding is the winner.

** Questions
- LLM was useful to generate questions with keywords and reference answers
- You need to manually check and update as they may have hallucinations

** Evaluation
- Prompt matters for better LLM evaluation with faithfulness and correctness.  Introducing CoT (Change of Thoughts) reduced hallucinations
- Still, you need to look at logs to see if LLM evaluation was correct

** LLM usage
- To stabilize results, setting temperature (& seed) is important
- I saw false positives and negatives.  LLMs are not reliable.  Reducing hallucinations is a challenge.  (eg, LLM could use its knowledge, not context to answer questions)

** Vector DB
- Transforming questions worsen the results.
